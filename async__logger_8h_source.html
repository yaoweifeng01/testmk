39 // Wrapper for a glog Logger which asynchronously writes log messages.
40 // This class starts a new thread responsible for forwarding the messages
41 // to the logger, and performs double buffering. Writers append to the
42 // current buffer and then wake up the logger thread. The logger swaps in
43 // a new buffer and writes any accumulated messages to the wrapped
44 // Logger.
45 //
46 // This double-buffering design dramatically improves performance, especially
47 // for logging messages which require flushing the underlying file (i.e WARNING
48 // and above for default). The flush can take a couple of milliseconds, and in
49 // some cases can even block for hundreds of milliseconds or more. With the
50 // double-buffered approach, threads can proceed with useful work while the IO
51 // thread blocks.
52 //
53 // The semantics provided by this wrapper are slightly weaker than the default
54 // glog semantics. By default, glog will immediately (synchronously) flush
55 // WARNING
56 // and above to the underlying file, whereas here we are deferring that flush to
57 // a separate thread. This means that a crash just after a 'LOG_WARN' would
58 // may be missing the message in the logs, but the perf benefit is probably
59 // worth it. We do take care that a glog FATAL message flushes all buffered log
60 // messages before exiting.
61 //
62 // NOTE: the logger limits the total amount of buffer space, so if the
63 // underlying
64 // log blocks for too long, eventually the threads generating the log messages
65 // will block as well. This prevents runaway memory usage.
